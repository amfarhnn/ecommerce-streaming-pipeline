# Real-Time E-Commerce Streaming Pipeline

An end-to-end data engineering project that simulates e-commerce user activity, streams events through Redpanda, processes them with a Python consumer, stores them in PostgreSQL, and exports analytics-ready datasets for dashboarding.

This project is designed to demonstrate a production-style streaming architecture in a compact local environment.

## What It Does

- Generates realistic e-commerce events such as `page_view`, `add_to_cart`, and `purchase`.
- Publishes events to a Kafka-compatible Redpanda topic.
- Consumes events in real time with a Python consumer group.
- Enriches each event with a `high_value` flag.
- Stores raw events in PostgreSQL.
- Maintains a live product sales summary table.
- Exports analytics CSVs that can be used in Power BI or another BI tool.

## Event Schema

Example event generated by the producer:

```json
{
  "event_id": "abc123",
  "user_id": 45,
  "user_name": "John Doe",
  "event_type": "purchase",
  "product": "Laptop",
  "price": 2999.0,
  "timestamp": "2026-04-30T10:30:00"
}
```

The consumer adds:

```json
{
  "high_value": 1
}
```

`high_value` is set to `1` when `price >= 1000`; otherwise it is set to `0`.

## Database Tables

### `events`

Stores every processed event.

| Column | Purpose |
| --- | --- |
| `event_id` | Unique event identifier |
| `user_id` | Simulated user id |
| `user_name` | Simulated user name |
| `event_type` | Event category |
| `product` | Product involved in the event |
| `price` | Event price |
| `timestamp` | Event creation time |
| `high_value` | Derived high-value transaction flag |

### `product_sales_summary`

Maintains a live aggregate for purchase events.

| Column | Purpose |
| --- | --- |
| `product` | Product name |
| `total_purchases` | Number of purchase events |
| `total_revenue` | Total revenue from purchases |

## Quick Start

### 1. Create the environment file

Create a `.env` file in the project root:

```env
POSTGRES_HOST=127.0.0.1
POSTGRES_DB=ecommerce_db
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres
POSTGRES_PORT=5433

KAFKA_BROKER=redpanda:9092
```

The Docker consumer overrides `POSTGRES_HOST` to `postgres` and `POSTGRES_PORT` to `5432` internally. The host port is published as `5433` to avoid conflicts with any local PostgreSQL installation already using `5432`.

### 2. Start the pipeline

```bash
docker compose up --build
```

This starts:

- `redpanda` on port `9092`
- `ecommerce_postgres` on host port `5433`
- `producer`, which emits one event every two seconds
- `consumer`, which processes and stores streamed events

### 3. Stop the pipeline

```bash
docker compose down
```

PostgreSQL data is stored in the `postgres_data` Docker volume, so it persists between runs.

## Local Python Setup

For running analytics scripts outside Docker:

```bash
pip install -r requirements.txt
```

Make sure Docker Compose is running first so PostgreSQL is available on `127.0.0.1:5433`.

For running the producer or consumer directly on the host, set `KAFKA_BROKER=127.0.0.1:9092` in your local environment. Inside Docker, keep `KAFKA_BROKER=redpanda:9092`.

## Reliability Notes

- Redpanda and PostgreSQL both use Docker health checks before the application services start.
- The consumer retries failed database writes up to three times.
- Event inserts use conflict handling so duplicate event ids are ignored.
- The sales summary updates only when a new purchase event is inserted, preventing duplicate Kafka replays from inflating aggregates.
